# Conflict Resolution: 충돌 해결 전략

> 관점 간 충돌을 트레이드오프 분석으로 해결

---

## 충돌 유형 분류

### Type 1: 우선순위 충돌

**정의:** 목표는 같으나 긴급도/중요도가 다름

**예시:**

```
기획자: "기능 A 우선" (비즈니스 가치)
개발자: "기술 부채 해결 우선" (안정성)
```

**해결 전략:**

1. 명확한 기한 설정
2. Phase 분할 (긴급 vs 중요)
3. 리소스 조정 (병렬 처리)

**트레이드오프 분석:**

```
옵션 A) 기능 A 우선
  장점: 빠른 비즈니스 가치
  단점: 기술 부채 누적
  리스크: 중장기 유지보수 비용 증가

옵션 B) 기술 부채 우선
  장점: 안정성 확보
  단점: 비즈니스 가치 지연
  리스크: 시장 기회 상실

옵션 C) 혼합 (권장)
  Week 1-2: 기술 부채 해결 (안정화)
  Week 3-4: 기능 A 개발 (빠른 구현)
  장점: 안정성 + 빠른 출시
  리스크: 낮음
```

---

### Type 2: 기술 vs 비즈니스 충돌

**정의:** 기술적 제약 vs 비즈니스 요구

**예시:**

```
비즈니스: "실시간 분석 필요"
개발자: "현재 스택으로 배치 처리만 가능"
```

**해결 전략:**

1. 기술 제약 명확히 설명 (비즈니스팀 이해)
2. 대안 기술 탐색 (개발팀 노력)
3. 스코프 조정 (타협)

**트레이드오프 분석:**

```
옵션 A) 실시간 분석 (비즈니스 요구)
  장점: 즉시 반응 가능
  단점: 아키텍처 재설계 필요 (3개월)
  리스크: 높음 (일정 지연)

옵션 B) 배치 처리 (현재 기술)
  장점: 즉시 구현 가능 (2주)
  단점: 1시간 지연
  리스크: 낮음

옵션 C) 단계적 접근 (권장)
  Phase 1: 배치 (1시간 지연, 2주)
  Phase 2: 준실시간 (5분 지연, 4주)
  Phase 3: 실시간 (인프라 업그레이드 후, 12주)
  장점: 빠른 검증 + 점진적 개선
  리스크: 중간
```

---

### Type 3: 보안 vs UX 충돌

**정의:** 보안 강화 vs 사용자 편의

**예시:**

```
보안: "2FA 필수"
UX: "간편 로그인 원함"
```

**해결 전략:**

1. 리스크 기반 접근 (상황별 차등)
2. 스마트 기본값 (편리하되 안전)
3. 사용자 선택권 (고급 설정)

**트레이드오프 분석:**

```
옵션 A) 2FA 필수 (보안 우선)
  장점: 높은 보안
  단점: 사용자 이탈 (20% 예상)
  리스크: UX 저하

옵션 B) 2FA 선택 (UX 우선)
  장점: 편리함
  단점: 보안 취약
  리스크: 계정 탈취

옵션 C) 리스크 기반 인증 (권장)
  Low risk: 비밀번호만 (조회, 설정)
  High risk: 2FA (결제, 개인정보 변경)
  장점: 보안 + UX 균형
  리스크: 낮음
```

---

### Type 4: 성능 vs 비용 충돌

**정의:** 성능 요구 vs 인프라 비용

**예시:**

```
기획자: "응답 1초 이하 필요"
인프라: "현재 인프라로 2초, 1초 위해 비용 2배"
```

**해결 전략:**

1. 성능 목표 수치화 (SLA)
2. 단계적 최적화 (측정 → 개선)
3. 비용 예산 설정

**트레이드오프 분석:**

```
옵션 A) 즉시 인프라 확장
  장점: 빠른 성능 확보
  단점: 비용 2배 (월 $1000 → $2000)
  리스크: 과도한 투자 (트래픽 불확실)

옵션 B) 현 상태 유지
  장점: 비용 절감
  단점: 성능 목표 미달
  리스크: 사용자 불만

옵션 C) 단계적 최적화 (권장)
  Phase 1: 소프트웨어 최적화 (2초 → 1.5초, 무료)
  Phase 2: 트래픽 측정 (2주)
  Phase 3: 필요시 선택적 확장 (병목만)
  장점: 비용 효율 + 성능 개선
  리스크: 낮음
```

---

### Type 5: 범위 vs 일정 충돌

**정의:** 기능 범위 vs 출시 일정

**예시:**

```
비즈니스: "모든 기능 필요"
개발자: "범위 축소 또는 일정 연장 필요"
```

**해결 전략:**

1. 핵심 기능 우선순위화 (MoSCoW)
2. MVP 정의 (80% 가치 제공)
3. Phase 분할 (빠른 출시 + 지속 개선)

**트레이드오프 분석:**

```
옵션 A) 전체 기능 (범위 우선)
  장점: 완성도 높음
  단점: 출시 3개월 지연
  리스크: 시장 변화, 경쟁사 선점

옵션 B) 최소 기능 (일정 우선)
  장점: 1개월 내 출시
  단점: 기능 부족 (사용자 불만 가능)
  리스크: 초기 리뷰 부정적

옵션 C) 핵심 기능 우선 (권장)
  Phase 1: Must-have (80% 가치, 1.5개월)
  Phase 2: Should-have (15% 가치, 1개월)
  Phase 3: Nice-to-have (5% 가치, 0.5개월)
  장점: 빠른 검증 + 지속 개선
  리스크: 낮음
```

## 합의 전략

### 전략 1: Win-Win 찾기 (우선)

**목표:** 양쪽 모두 만족하는 해결책

**방법:**

```
1. 각 관점의 핵심 우려사항 파악
   기획자: 빠른 출시
   개발자: 품질 보장

2. 공통 목표 찾기
   → 성공적인 제품 출시

3. 양립 가능한 방법 탐색
   → Phase 분할: 빠른 MVP + 점진적 개선
```

**성공률:** 60%

**예시:**

```
기획자 vs 개발자 (개발 기간)

기획자 우려: 시장 선점
개발자 우려: 품질 보장

Win-Win:
- Phase 1: 핵심 기능 (2주, 시장 선점)
- Phase 2: 고급 기능 (1주, 품질 보장)
- 결과: 양쪽 모두 만족
```

---

### 전략 2: 조건부 합의 (차선)

**목표:** 특정 조건 충족 시 합의

**방법:**

```
1. 합의 조건 명확히 정의
   "테스트 커버리지 80% 이상이면 출시"

2. 조건 달성 계획 수립
   단위 테스트: 2일
   통합 테스트: 1일

3. 조건 검증 방법 합의
   → pytest --cov 실행, 리포트 공유
```

**성공률:** 30%

**예시:**

```
보안 vs 개발자 (출시 일정)

보안: "보안 리뷰 없이 출시 불가"
개발자: "일정상 리뷰 시간 부족"

조건부 합의:
- 조건: 자동 보안 스캔 100% 통과
- 방법: SonarQube + SAST 실행
- 검증: 리포트 자동 생성, 보안팀 승인
- 결과: 개발 일정 유지 + 보안 보장
```

---

### 전략 3: 사용자 결정 위임 (최후)

**목표:** 트레이드오프 명확히 제시 후 사용자 결정

**방법:**

```
1. 충돌 상황 명확히 설명
2. 각 옵션의 장단점 객관적 제시
3. 권장사항 제시 (근거 포함)
4. AskUserQuestion 호출
```

**사용 조건:**

- 양쪽 모두 Hard Constraint (타협 불가)
- 비즈니스 판단 필요
- 리스크 수용 여부 결정

**예시:**

```
보안 vs 성능 (암호화)

보안: "개인정보 암호화 필수" (법적 요구)
성능: "암호화 시 20% 성능 저하"

AskUserQuestion:
"개인정보 암호화는 법적 필수사항이나, 성능이 20% 저하됩니다.

옵션:
A) 암호화 적용 (법 준수, 성능 저하 수용)
   장점: 법적 준수, 보안 강화
   단점: 응답시간 1초 → 1.2초
   리스크: 사용자 경험 저하 (일부)

B) 인프라 확장 (암호화 + 성능 유지, 비용 증가)
   장점: 법 준수 + 성능 유지
   단점: 월 비용 +$500
   리스크: 예산 초과

C) 개인정보 수집 최소화 (암호화 대상 축소)
   장점: 성능 + 비용 유지
   단점: 기능 제약 (일부)
   리스크: 제품 가치 감소

권장: B (인프라 확장)
근거: 법적 요구 + 사용자 경험 보장"
```

## 합의 수준 분류

### ✅ 전원 합의 (Unanimous)

**정의:** 모든 관점이 동의

**예시:**

```
"P0 질문 필수" → 기획자, 개발자, 보안 전원 동의
"트랜잭션 무결성" → 개발자, 보안, 데이터 전원 동의
```

**신뢰도:** 매우 높음

**행동:** 즉시 진행

---

### ⚠️ 조건부 합의 (Conditional)

**정의:** 특정 조건 충족 시 합의

**예시:**

```
"출시 가능" (조건: 테스트 커버리지 80% 이상)
"Phase 2 연기 가능" (조건: 모니터링 강화)
```

**신뢰도:** 높음

**행동:** 조건 달성 계획 수립

---

### ❓ 사용자 결정 필요 (Pending)

**정의:** 관점 간 타협 불가, 사용자 판단 필요

**예시:**

```
"비용 vs 성능" (비즈니스 판단)
"범위 vs 일정" (우선순위 결정)
```

**신뢰도:** 중간

**행동:** AskUserQuestion 호출, 답변 대기

---

### ❌ 미합의 (No Consensus)

**정의:** 합의 실패

**예시:**

```
"근본적 설계 충돌" (아키텍처 재검토 필요)
```

**신뢰도:** 낮음

**행동:** 상위 의사결정권자 에스컬레이션 또는 설계 재검토

---

## 충돌 해결 체크리스트

### 분석 단계

- [ ] 충돌 근본 원인 파악 (정보 부족? 목표 불일치? 제약 충돌?)
- [ ] 각 관점의 핵심 우려사항 정리
- [ ] Hard Constraint 식별 (양보 불가능한 요소)

### 해결 단계

- [ ] Win-Win 해결책 탐색 (우선)
- [ ] 조건부 합의 가능성 검토 (차선)
- [ ] 트레이드오프 명확히 분석 (최후)

### 검증 단계

- [ ] 양쪽 우려사항 해소 확인
- [ ] 잔여 리스크 식별 및 완화 방안
- [ ] 합의 수준 명시 (전원/조건부/사용자 결정)

## 실전 팁

### 팁 1: 충돌을 숨기지 않는다

```
❌ "개발자와 조율하여 해결했습니다" (모호)
✅ "개발 기간 충돌 → Phase 분할로 합의 (상세 내역)"
```

### 팁 2: 트레이드오프를 명확히 한다

```
❌ "A가 더 좋습니다"
✅ "A는 X 장점, Y 단점. B는 Z 장점, W 단점. 상황에 따라 A 권장"
```

### 팁 3: 합의 과정을 투명하게 한다

```
Round 1: 충돌 발견
Round 2: 각 관점 재검토
Round 3: 트레이드오프 분석 → 합의안 도출
```

### 팁 4: 결정 근거를 기록한다

```
합의안: Phase 분할
근거:
- 기획자 우려 해소: 2주면 시장 선점 가능
- 개발자 우려 해소: 핵심 기능 품질 보장
- 리스크 완화: MVP로 빠른 검증 후 개선
```
