# 리스크 분석 가이드

## 리스크 분류

### 기술적 리스크

| 유형 | 예시 | 대응 |
|------|------|------|
| **의존성 충돌** | 라이브러리 버전 불일치 | 버전 고정, 테스트 |
| **성능 저하** | N+1 쿼리, 메모리 누수 | 성능 테스트, 프로파일링 |
| **호환성 문제** | 브라우저 지원, API 변경 | 폴리필, 버전 체크 |

### 비즈니스 리스크

| 유형 | 예시 | 대응 |
|------|------|------|
| **요구사항 변경** | 기획 수정 | 모듈화, 유연한 설계 |
| **데이터 무결성** | 잘못된 계산, 중복 처리 | 검증 로직, 트랜잭션 |
| **보안 취약점** | 인증 우회, 데이터 노출 | 보안 리뷰, 테스트 |

## 리스크 평가 매트릭스

### 가능성 (Likelihood)
- **High**: 자주 발생 또는 발생 확실
- **Medium**: 가끔 발생 가능
- **Low**: 거의 발생하지 않음

### 영향도 (Impact)
- **High**: 서비스 중단, 데이터 손실
- **Medium**: 기능 저하, 사용자 불편
- **Low**: 미미한 영향

### 우선순위
| 가능성 \ 영향도 | High | Medium | Low |
|-----------------|------|--------|-----|
| **High** | 🔴 Critical | 🟠 High | 🟡 Medium |
| **Medium** | 🟠 High | 🟡 Medium | 🟢 Low |
| **Low** | 🟡 Medium | 🟢 Low | 🟢 Low |

## 대응 전략 유형

### 회피 (Avoid)
리스크를 발생시키지 않도록 계획 변경
- 예: 불안정한 라이브러리 대신 검증된 대안 사용

### 완화 (Mitigate)
리스크 발생 가능성 또는 영향 감소
- 예: 단계별 롤아웃, 기능 플래그

### 전가 (Transfer)
리스크 책임을 다른 곳으로 이동
- 예: 외부 서비스 사용, 보험

### 수용 (Accept)
리스크를 인지하고 감수
- 예: 영향이 작은 UI 깨짐
